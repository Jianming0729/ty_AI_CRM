# 横向治理模块 (Governance Sync) 工作流 (V3.1)

本工作流定义了“修复”与“治理”的权责边界，确保系统在保持灵活修复的同时，严控经验沉淀的安全质量。

## 一、模式判定原则 (M0)

1. **默认模式：修复 / 开发模式 (Fix Mode)**
   - 目标：快速迭代，修复线上问题。
   - 约束：仅限代码修复及日志增强。
   - **授权：无需人类书面签字**。

2. **触发模式：治理同步模式 (Governance Sync Mode)**
   - 触发词：“治理同步”、“成功经验沉淀”、“更新 README”等。
   - 目标：将稳定后的逻辑固化为长期资产。
   - **授权：必须检测到人类 G0 级书面确认**。

## 二、执行步骤

### 状态 1：修复阶段 (Fixing)
- 直接执行代码修改、测试与部署。
- 不得宣称“已彻底解决”。

### 状态 2：治理申请 (Applying)
- 运行提示词：
  > “本次对【功能 / 模块名】的修改已完成并通过测试，申请进入‘横向治理模块（Governance Sync）’待确认状态。”

### 状态 3：人工授权 (Authorize)
- 人类提供书面指令（G0）：
  > “本人已人工确认：该功能在真实使用场景下稳定可用，问题已实质性解决，允许执行一次横向治理模块受控同步。”

### 状态 4：经验沉淀 (Precipitating)
- 依照 `docs/governance_sync_v3.md` 执行文档同步与经验记录。

## 三、行为边界

- **Fix Mode**：能写代码，不能改 README。
- **Governance Mode**：能改 README，不能再试错。
- **G0 闸门**：仅守护“治理同步模式”。
